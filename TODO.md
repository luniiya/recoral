  - **Promoted to top priority**: this is a personal-necessity feature for the user, whose own Google Recorder app is bricked from too many recordings, not just a nice-to-have for other users.
  - **Format reverse-engineered (2026-07-13)** against the user's own real Takeout export (1000 recordings, 273MB, received by email, extracted for inspection to a scratch dir outside the repo, original `.zip` kept at the repo root but gitignored, never committed):
    - Structure is completely flat: `Takeout/Recorder/*.m4a` plus, for some recordings, a matching `Takeout/Recorder/<same base name>_summary.txt`. No manifest/index/JSON file anywhere in the export.
    - Only 294 of the 1000 `.m4a` files have a matching `_summary.txt`, Google only auto-transcribed some recordings, not all. Don't assume every audio file has a transcript.
    - **Filenames are not a reliable date source.** Google's auto-generated name is `"D Mon at HH_MM"` (e.g. `22 Nov at 19_40.m4a`), no year at all, and only minute resolution, so two recordings made in the same minute collide; Google itself resolves that by appending `(1)`, `(2)`, etc. (293 of the 1000 files have this suffix). User-renamed recordings (e.g. `vM kinda ASMR successful .m4a`, `anya haha inakute samishii.m4a`) have no date in the filename whatsoever.
    - **The real creation date is embedded in the `.m4a` container itself**, not the filename: `creation_time` (full UTC timestamp, e.g. `2026-05-10T16:58:03.000000Z`), readable via `ffprobe -show_format`. This is the authoritative source for the recording's date/time on import. `ffprobe`'s `duration` field gives the real duration too. Spot-checked several against expected dates and they're credible.
    - **Title mapping decided**: if a filename matches Google's auto-generated `"D Mon at HH_MM"` pattern (with or without a `(N)` collision suffix), treat it as no title on import, store it blank and let recoral's existing "blank title shows the date instead" convention handle display, exactly like a recording made natively in recoral with no title. If the filename doesn't match that pattern, it's a real user-given title, import it as-is.
    - **No separate "description" field exists anywhere in the export**, despite Google Recorder's app allowing you to add one. Checked exhaustively across all 1000 files: format-level tags are always exactly `major_brand`, `minor_version`, `compatible_brands`, `creation_time`, `com.android.version` (never more); stream-level tags are always exactly `creation_time`, `handler_name`, `language`; every `.txt` file is the same `_summary` type, no `_description.txt` variant exists. Conclusion: Takeout just doesn't include descriptions for Recorder, not a gap in our parsing. Don't build for a field that isn't there.
    - Each `.m4a` also has three extra `mett` (Android timed-metadata) data streams beyond the audio stream itself. Not investigated further, not worth chasing since the separate `_summary.txt` already gives a clean, readable transcript.
    - **Transcript format**: when a `_summary.txt` exists (despite the filename, it's a real per-segment transcript, not a one-line summary), it's repeating blocks of a timestamp line (`MM:SS`, or `HH:MM:SS` for longer recordings) followed by a paragraph of transcribed text, blank-line separated. Some transcripts additionally prefix each timestamp with a speaker label, e.g. `Speaker 1 00:00`, `Speaker 2 00:10`, this is Google Recorder's built-in speaker-recognition feature, and the user has confirmed it doesn't work well/at all in practice. **Decision: don't build dedicated speaker-aware storage for this.** On import, flatten the transcript to plain text (keep the `Speaker N` prefix inline as part of the text if present, don't try to parse/structure it into per-speaker data) and store it in the existing flat `transcript` column. Not worth trusting or preserving structurally given the source feature is known-unreliable.
    - No language field found either; transcripts are a mix of languages (at least English and French seen in samples) depending on what was actually spoken.
  - Plan (not fully decided, but converged on this shape): a dedicated Import/Export section in Settings, drag-and-drop (or file picker) upload of the Takeout archive, server-side processing after upload (not client-side, these can be large). The backend needs to process the archive in a memory-safe/streaming way (stream-unzip and import recordings one at a time or in small batches, not load the whole archive into memory at once) rather than assuming small/typical upload sizes like the existing single-file `POST /api/recordings` path does, this is not a hypothetical concern, the user's own real export is 1000 recordings. Each extracted recording should go through the same creation path a normal upload does (duplicate content-hash check, storage quota enforcement), just fed a pre-known title/date/transcript instead of deriving them from the upload request. If a recording ends up transcript-less from Takeout, it should still be eligible for recoral's own transcription pipeline (see Transcription section below) rather than being permanently blank.

  - **Refinement over the original plan's "API shape" note**: instead of whisper.cpp's own bundled server (which loads one fixed model for its whole process lifetime), `transcription/server.ts` is a thin Bun HTTP wrapper that shells out to the `whisper-cli` binary per request, picking the model file per call. Recordings still `pending`/`processing` at server boot (crash, restart) get requeued automatically. Shared between new uploads and Takeout/recoral-import backfill (both `takeoutImport.ts` and `recoralImport.ts` enqueue any recording that arrives without a transcript already, since Takeout only ships a transcript for some recordings, not all).

A failed recording almost never succeeds on a bare retry with no other change, so `listUntranscribed()` now only matches `'none'`, and a failed recording stays failed until the user explicitly hits Retry (which already calls `enqueueTranscription()` directly, bypassing this sweep). Verified against a real server boot: a recording manually marked `failed` stayed `failed` across a restart, one marked `none` correctly flipped to `pending`.

  - **Real bugs found and fixed against the actual live deployment (2026-07-17), all in Bun v1.3.x**: (1) `form.append("file", Bun.file(path), name)` deadlocks forever once that `FormData` reaches `fetch()`. (2) `AbortSignal.timeout(ms)` fires immediately instead of after `ms`. Fixed with a manual `AbortController` + `setTimeout`. (3) The actual root cause of transcription never completing at all: `fetch()` has a hardcoded ~5-minute socket timeout that fires regardless of any `AbortSignal`, indistinguishable from a real abort, fatal since CPU-only whisper takes 10-15 minutes per clip. Fixed by dropping `fetch()` for this call in favor of `node:http`'s `http.request()` (no such built-in timeout), with a manual 30-minute safety limit via `setTimeout` + `req.destroy()`. Also hardened `processQueue()`: two DB calls used to sit outside the per-item `try/catch`, so one unexpected error silently killed the *entire remaining queue* with no explanation (confirmed: happened overnight, 215 recordings frozen until someone noticed), both calls are now inside the try, plus a defensive catch around the whole per-item call in the loop.
  - **Earlier bug (2026-07-15, sandbox pipeline test, still valid)**: `Bun.write(path, response)` in `transcription/server.ts`'s model-download code hung forever; `Bun.write(path, response.body)` resolved fast but silently wrote the literal string `"[object ReadableStream]"` instead of the stream's bytes. Fixed by manually pumping `res.body` through `Bun.file(path).writer()` chunk-by-chunk (verified against `Content-Length`), plus an atomic `rename()` for the temp-file finalization instead of a redundant full copy.
  - **Still open**: no GPU/CPU hardware detection or fallback (`Dockerfile` is unconditionally CPU-only now, no CUDA/ROCm path exists); the NPM reverse-proxy port on the real deployment needs a manual fix (was pointed at the host-exposed `3320`, needs the container's actual internal port, `7245` after this session's port change); leftover `/tmp/transcribe-*/` dirs from old timed-out requests need a manual cleanup on that server; `docker-compose.yml`'s `network: host` build-context workaround for that server's own network setup was deliberately not brought into this repo since it's specific to that host, not general.
- [ ] **Word-level timestamps for real-time highlight-while-playing** (explicitly flagged as an important *future* feature, not a current priority, "we'll figure it out on time"):
  - Would reuse playback-position infrastructure already built this session: `RecordingDetail` already tracks live `audioEl.currentTime` (built for the waveform's playhead sync). The transcript view would just need to find whichever word's `[start, end]` window contains the current time, highlight it, and auto-scroll to keep it in view.
  - Storage format deliberately left undecided: a naive `[{word, start, end}, ...]` JSON array was floated but flagged as wasteful (repeated key names + braces/commas per word, adds up across a whole transcript). A more compact array-of-arrays shape (e.g. `[[word, startMs, endMs], ...]`, no repeated keys) was suggested as a likely better fit whenever this is actually built, but not committed to.
  - Whisper/whisper.cpp's native word timestamps (DTW/cross-attention based) are usable but can be jittery, especially on fast speech or non-English audio; more accurate forced-alignment approaches (WhisperX-style, a separate per-language phoneme model) exist but add real complexity (multilingual alignment models needed, since the user's own recordings mix languages). Start with whisper.cpp's built-in word timestamps; only reach for forced alignment later if sync quality is actually a problem in practice.

- [ ] Decide if local/on-device transcription is ever supported in offline mode, or always server-only
  - Confirmed technically feasible via phone NPU: iOS `SFSpeechRecognizer` with `requiresOnDeviceRecognition = true` (built-in, free, Apple maintains the model), Android via ML Kit on-device speech-to-text or a bundled quantized Whisper (`whisper.cpp`/`whisper.tflite`, `tiny`/`base` models run fast on modern NPUs). Precedent: Google's own Pixel Recorder app already transcribes fully on-device via the Tensor NPU.
  - Real-time word-by-word streaming isn't required, fast on-device batch transcription right after recording covers the voice-memo use case fine and is much simpler.
  - Needs a custom Capacitor native plugin (Swift + Kotlin) bridging to the platform ASR API or bundled model runtime, not reachable from the WebView/JS layer directly. Same shape of native work as the `capacitor-audio-recorder` plugin already planned for background recording, but a second plugin specific to transcription.
  - On-device is the only way fully-offline-mode users (no account, no server, ever) get transcripts at all, fits the "offline is first-class" principle hard. But it doesn't help the desktop webUI (no NPU access from a browser), so a server-side transcription path is still needed regardless, meaning two pipelines to maintain if both are built, not one.
